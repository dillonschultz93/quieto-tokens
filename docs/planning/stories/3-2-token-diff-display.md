# Story 3.2: Token Diff Display

Status: review

## Story

As a **solo developer making changes to my token system**,
I want to see exactly what will change before any files are written,
so that I can understand the impact of my modifications and catch unintended consequences.

## Story Scope Note

This story enhances the `quieto-tokens update` command (shipped in Story 3.1) with a **token diff display** that appears between the selective regeneration pipeline and the confirm/write step. Today, `update` runs `previewAndConfirm` which renders the entire final token system — the user sees all tokens but has no visibility into what actually changed vs. what was preserved from disk. This story adds a focused diff view that groups changes by tier and category, distinguishes added/modified/removed tokens, and traces cascading impacts (e.g., "Changing `color.blue.500` affects 12 semantic tokens").

**What this story IS:**
- A diff engine that compares the prior on-disk token state against the pipeline's regenerated collection.
- A diff renderer that displays changes in the terminal, grouped by tier (primitives → semantics), with color swatches for color token diffs and distinct markers for added/modified/removed tokens.
- A cascade summary that traces how primitive changes ripple into semantic tokens.
- Integration into the `update` command's flow, shown before the existing preview + override step.
- A "no changes detected" early-exit path when the regenerated tokens are identical to the on-disk tokens.

**What this story is NOT:**
- Not `--dry-run` (Story 3.3 — this story's diff only appears in the `update` flow, not `init`/`add`/`component`).
- Not changelog generation (Story 3.4 — the diff is a terminal-only display, not persisted).
- Not a modification to the `init` modify-flow — `init` continues to use the full preview without a diff.
- Not component-token diffing — component tokens are pure references and are not regenerated by `update` (Story 3.1 AC #19).

## Acceptance Criteria

### Diff computation

1. **Given** the `update` pipeline has produced a regenerated `ThemeCollection`, **When** the diff step runs, **Then** the prior token state is loaded from disk using `loadPrimitivesFromDisk` and `loadSemanticTokensFromDisk` (from Story 3.1's `src/pipeline/load-from-disk.ts`), and a token-level diff is computed.
2. **Given** the diff is computed, **When** tokens are compared, **Then** each token is matched by its canonical path (e.g., `color.blue.500`, `spacing.md`). A token present in the regenerated collection but absent on disk is **added**. A token present on disk but absent in the regenerated collection is **removed**. A token present in both but with a different `$value` is **modified**.
3. **Given** no tokens have changed (regenerated collection is identical to disk), **When** the diff is computed, **Then** the diff result is empty and the CLI reports "No changes to apply — your token system is up to date." via `p.log.info` and exits cleanly without proceeding to preview or writing files.

### Diff display — primitives

4. **Given** primitive tokens have changed, **When** the diff display renders, **Then** changes are grouped by category (e.g., "Color Primitives", "Spacing Primitives"), and within each category, grouped by subcategory where applicable (e.g., by hue for color tokens).
5. **Given** a color primitive was modified, **When** its diff line renders, **Then** both the old and new hex values are displayed with ANSI color swatches (where terminal supports it): e.g., `██ color.blue.500  #2563eb → ██ #1d4ed8`. Use `hexToAnsi` from `src/utils/color-display.ts`.
6. **Given** a non-color primitive was modified (spacing, typography), **When** its diff line renders, **Then** the old and new values are displayed inline: e.g., `spacing.md  16px → 24px`.
7. **Given** a primitive was added (e.g., a new hue ramp), **When** its diff line renders, **Then** it is prefixed with a `+` marker and the value is shown: e.g., `+ color.teal.500  #14b8a6`.
8. **Given** a primitive was removed (e.g., a hue ramp was dropped), **When** its diff line renders, **Then** it is prefixed with a `−` marker and the old value is shown: e.g., `− color.teal.500  #14b8a6`.

### Diff display — semantics

9. **Given** semantic tokens have changed, **When** the diff display renders, **Then** changes are grouped by theme (e.g., "Semantic Changes (Default)", "Semantic Changes (Light)", "Semantic Changes (Dark)"), and within each theme, grouped by category.
10. **Given** a semantic token's `$value` reference changed (e.g., `{color.blue.500}` → `{color.blue.600}`), **When** its diff line renders, **Then** both the old and new reference are shown: e.g., `color.background.primary  → {color.blue.500} → {color.blue.600}`. For color semantics, include swatches resolved from the primitive palette.
11. **Given** a semantic token was added or removed, **When** its diff line renders, **Then** the same `+`/`−` marker convention from primitives applies.

### Cascade tracing

12. **Given** one or more primitive tokens changed, **When** the diff step completes, **Then** a cascade summary is displayed showing how many semantic tokens were affected: e.g., "Changing 3 color primitives affects 12 semantic tokens across 2 themes." The summary appears after the detailed diff, before the preview.
13. **Given** a primitive was removed AND semantic tokens still reference it, **When** the cascade summary renders, **Then** the affected semantic tokens are called out as broken references: e.g., "⚠ color.background.accent still references removed primitive {color.teal.500}". (This is separate from Story 3.1's override conflict resolution, which handles user overrides — this catches mapper-default references that became stale.)

### User flow integration

14. **Given** the diff has been displayed, **When** the user is prompted, **Then** they see a `p.select` with options: "Accept changes and write", "Review full token preview" (enters existing `previewAndConfirm`), "Go back and modify further" (returns to category picker), or "Cancel" (exit without writing).
15. **Given** the user selects "Accept changes and write", **When** the write step runs, **Then** the output is written via `runOutputGeneration` with the scoped `WriteScope` (same as Story 3.1 — only modified categories' files are written). The existing preview step is skipped.
16. **Given** the user selects "Review full token preview", **When** the preview runs, **Then** the existing `previewAndConfirm` flow is entered (showing all tokens, offering the override loop). After the preview, the user can accept or cancel.
17. **Given** the user selects "Go back and modify further", **When** the flow resumes, **Then** the category picker from Story 3.1's `collectUpdateInputs` is re-entered. The user can select additional categories to modify. After making changes, the pipeline re-runs and a new diff is displayed.
18. **Given** the user selects "Cancel", **When** the cancel is processed, **Then** no files are written and the config is unchanged.

### Edge cases

19. **Given** the user runs `update` and makes changes that result in the exact same token values (e.g., re-enters the same brand color hex), **When** the diff is computed, **Then** the diff is empty and the "No changes to apply" early-exit activates (AC #3).
20. **Given** add-on categories exist on disk (e.g., shadow, border) but were not modified, **When** the diff displays, **Then** those categories are not shown in the diff at all — only modified categories appear.
21. **Given** component tokens exist, **When** the diff displays, **Then** component tokens are not diffed (they are not regenerated by `update`). A note is shown: "Component tokens are unchanged — they reference semantic tokens and will inherit your changes via the CSS cascade."

### Previous Story Intelligence (Story 3.1)

Story 3.1 shipped the `update` command with selective regeneration and `loadPrimitivesFromDisk` / `loadSemanticTokensFromDisk` / `loadComponentTokensFromDisk` in `src/pipeline/load-from-disk.ts`. These loaders reconstruct `PrimitiveToken[]`, `Map<string, SemanticToken[]>`, and `ComponentToken[]` from the on-disk DTCG JSON files using a `walkDtcgLeaves()` callback pattern.

Key patterns from Story 3.1 to build on:
- `updateCommand` (in `src/commands/update.ts`) orchestrates: load config → collect inputs → run pipeline → resolve override conflicts → preview → write. The diff display inserts between "resolve conflicts" and "preview".
- `UpdatePipelineResult` contains `{ collection: ThemeCollection; modifiedCategories: readonly string[] }`. The `modifiedCategories` list determines which categories are interesting for diffing (AC #20: only show changed categories).
- `previewAndConfirm` (in `src/ui/preview.ts`) renders the full collection and offers the override loop. Story 3.2 makes the full preview an opt-in step rather than the default — the diff is the new default view for `update`.
- `src/utils/color-display.ts` exports `hexToAnsi(hex)` and `supportsColor()` — reuse for diff color swatches.

### Diff Architecture

```
Prior collection (from disk)     Current collection (from pipeline)
         │                                    │
         └──────────┬─────────────────────────┘
                    │
           computeTokenDiff()
                    │
              TokenDiff {
                primitiveChanges: TokenChange[]
                semanticChanges: Map<themeName, TokenChange[]>
                isEmpty: boolean
              }
                    │
         ┌──────────┼──────────┐
         │          │          │
  renderTokenDiff  renderCascade  (no changes → exit)
         │          │
         └──────────┘
              │
    Post-diff p.select prompt
    ┌────┬────┬────┬────┐
  Accept Preview Back Cancel
```

### Technical Constraints

- **Diff engine is pure.** `computeTokenDiff` takes two `ThemeCollection`s and returns a `TokenDiff`. No I/O, no Clack. Trivially testable.
- **Renderer is Clack-based.** `renderTokenDiff` and `renderCascadeSummary` call `p.log.step`, `p.log.info`, `p.log.warn`. Tested by mocking `@clack/prompts` (same pattern as `src/ui/__tests__/preview.test.ts`).
- **Prior collection loading reuses Story 3.1 loaders.** The `loadPriorCollection` helper in `src/pipeline/diff-loader.ts` is a thin composition of the existing `loadPrimitivesFromDisk`, `loadSemanticTokensFromDisk`, and `loadComponentTokensFromDisk`.
- **Scoping diff to modified categories.** While `computeTokenDiff` compares full collections, `renderTokenDiff` can filter its output to only show changes in `modifiedCategories`. This keeps the display focused — the user doesn't see noise from categories they didn't touch (whose loaded-from-disk values should be identical to the regenerated values).

### What NOT to Build

- **Do NOT implement `--dry-run`.** That is Story 3.3.
- **Do NOT implement changelog generation.** That is Story 3.4. The diff is terminal-only.
- **Do NOT add diff display to `init`, `add`, or `component`.** Story 3.2 scopes the diff to `update` only. Other commands continue to use the full preview. Extending diff to other commands can be revisited in a future story.
- **Do NOT implement interactive per-token accept/reject.** The user accepts or rejects the whole change set. Fine-grained overrides are available through the existing `previewAndConfirm` override flow.
- **Do NOT diff component tokens.** Components are pure references, not regenerated by `update`. The note about CSS cascade inheritance is sufficient.
- **Do NOT diff CSS output.** The diff operates at the DTCG token level, not at the CSS custom property level. CSS is a build artifact rebuilt from tokens.
- **Do NOT persist the diff.** The diff is a terminal display. Persistence is Story 3.4 (changelog).
- **Do NOT migrate `compareVersions`, extract the shared version resolver, or add lockfile protection.** All still deferred per `docs/planning/deferred-work.md`.
