# Story 1.6: Theme Variant Generation

Status: done

## Story

As a **solo developer**,
I want light and dark theme variants generated automatically from my palette,
So that I support both appearances without designing two separate color systems.

## Acceptance Criteria

1. **Given** the user selected "yes" to light/dark theme generation (from Story 1.2), **When** the theme generation step runs, **Then** a light theme is generated with appropriate semantic mappings (light backgrounds, dark text).
2. **Given** a light theme is generated, **When** dark theme generation runs, **Then** a dark theme is generated by inverting the semantic mappings (dark backgrounds, light text).
3. **Given** both themes are generated, **When** the theme tokens are examined, **Then** both themes reference the same primitive palette — only the semantic mappings differ.
4. **Given** the user selected "no" to themes (from Story 1.2), **When** the theme step runs, **Then** only a single default theme is generated (the light mapping from Story 1.5 becomes the sole theme).
5. **Given** themes are generated, **When** the output structure is prepared, **Then** the theme structure follows Style Dictionary's platform/theme conventions for later processing in Story 1.8.

### Architecture Constraints

- **ESM-only** — all imports use ESM syntax.
- **Themes are remappings, not regenerations:** Both light and dark themes use the SAME primitive palette. Only the semantic token `$value` references change. This is the core architectural principle.
- **Style Dictionary v5 is the target consumer:** The theme data model must be structured for Style Dictionary to process in Story 1.8. Research SD v5's multi-theme patterns BEFORE finalizing the data model.
- **Clack for all output** — `log.step()` / `log.info()` for narrative.

### Theme Architecture

The in-memory structure after this story completes:

```
ThemeCollection {
  primitives: PrimitiveToken[]          // Shared — color, spacing, typography
  themes: [
    {
      name: "light",
      semanticTokens: SemanticToken[]   // Same as Story 1.5 output
    },
    {
      name: "dark",
      semanticTokens: SemanticToken[]   // Inverted color mappings, same spacing/typo
    }
  ]
}
```

If the user chose no themes, the collection has a single theme named "default" with the standard mappings.

### Dark Mode Inversion Strategy

The inversion is a **ramp step flip**, not a color value recalculation:

| Semantic Role | Light Theme Ref | Dark Theme Ref |
|--------------|-----------------|----------------|
| `color.background.default` | `color.neutral.50` (lightest) | `color.neutral.900` (darkest) |
| `color.background.primary` | `color.blue.500` | `color.blue.400` (slightly lighter for dark bg contrast) |
| `color.content.default` | `color.neutral.900` (darkest) | `color.neutral.50` (lightest) |
| `color.content.primary` | `color.blue.700` | `color.blue.300` |
| `color.border.default` | `color.neutral.200` | `color.neutral.700` |

The exact step mapping depends on the ramp steps available from `@quieto/palettes`. The key principle: maintain sufficient contrast between background and content tokens in both themes.

### Style Dictionary v5 Theme Conventions

Style Dictionary v5 supports multiple approaches for theming:
1. **Separate source directories** per theme: `tokens/semantic/light/`, `tokens/semantic/dark/`
2. **Platform-level filtering** using `include`/`source` arrays per theme
3. **DTCG `$extensions`** with mode metadata

The recommended approach for this CLI: **separate source directories per theme** with shared primitives. This is the most straightforward for both SD processing and human readability:
```
tokens/
├── primitive/           # Shared across all themes
│   ├── color.json
│   ├── spacing.json
│   └── typography.json
├── semantic/
│   ├── light/           # Light theme semantic mappings
│   │   ├── color.json
│   │   ├── spacing.json
│   │   └── typography.json
│   └── dark/            # Dark theme semantic mappings
│       ├── color.json
│       ├── spacing.json
│       └── typography.json
```
