# Contributing to Quieto Tokens

Thanks for your interest in contributing! This guide covers everything you need to get started.

## Prerequisites

- [Node.js](https://nodejs.org/) 18 or newer
- npm (ships with Node)

## Getting Started

1. Fork and clone the repository:

```bash
git clone https://github.com/<your-username>/quieto-tokens.git
cd quieto-tokens
```

2. Install dependencies:

```bash
npm install
```

3. Build the project:

```bash
npm run build
```

4. Run the test suite to make sure everything is working:

```bash
npm test
```

## Development Workflow

### Useful Commands

| Command | Description |
|---|---|
| `npm run build` | Production build via tsup |
| `npm run dev` | Watch mode — rebuilds on file changes |
| `npm run type-check` | Run `tsc --noEmit` to check types without emitting |
| `npm test` | Run the full test suite (Vitest) |
| `npm run test:watch` | Run tests in watch mode |

### Project Structure

```
src/
  cli.ts              # CLI entry point
  index.ts            # Library entry point
  commands/           # CLI command handlers (init, add, component)
  generators/         # Token generation logic per category
  mappers/            # Semantic token mapping
  output/             # File writers (JSON, CSS via Style Dictionary)
  pipeline/           # Orchestration layer
  types/              # Shared TypeScript types
  ui/                 # Terminal UI (preview, prompts)
  utils/              # Shared utilities
  __tests__/          # Top-level integration tests
```

Tests live in `__tests__/` directories colocated with the code they cover (e.g., `src/generators/__tests__/color.test.ts`).

### Running a Local Build

To test the CLI end-to-end during development:

```bash
npm run build
npm link
quieto-tokens init
```

## Making Changes

### Branching

Create a feature branch off `main`:

```bash
git checkout -b your-branch-name main
```

If your change relates to a specific story or issue, include the identifier in the branch name (e.g., `2-3-guided-component-token-generation`).

### Commits

This project follows [Conventional Commits](https://www.conventionalcommits.org/). Format your commit messages as:

```
<type>(<scope>): <description>
```

Common types:

- `feat` — new feature or capability
- `fix` — bug fix
- `docs` — documentation only
- `refactor` — code change that neither fixes a bug nor adds a feature
- `test` — adding or updating tests

Scope is optional but encouraged. Use the area of the codebase you're touching (e.g., `cli`, `animation`, `pipeline`, `planning`).

Examples:

```
feat(cli): add border category to add subcommand
fix(animation): ease path collision
docs(planning): mark story 2-3 as done
```

### Tests

All new functionality should include tests. Tests use [Vitest](https://vitest.dev/) and follow this convention:

- Place test files in a `__tests__/` directory next to the source code
- Name test files `<module>.test.ts`
- Run `npm test` before pushing to make sure nothing is broken

### Type Checking

The project uses TypeScript in strict mode. Run `npm run type-check` to catch type errors before submitting.

### Generated Files

Do **not** hand-edit files in `tokens/` or `build/`. These are generated by the CLI and carry a `$metadata.doNotEdit` banner. Edit the generation logic in `src/` instead.

## Submitting a Pull Request

1. Make sure all tests pass (`npm test`) and types check (`npm run type-check`)
2. Push your branch to your fork
3. Open a pull request against `main`
4. Describe what your PR does and why — link to the relevant issue or story if one exists

PRs are squash-merged. Keep your commits clean, but don't stress about a perfect history — the merge commit message is what matters.

## Reporting Bugs and Requesting Features

Open an issue at [github.com/dillonschultz93/quieto-tokens/issues](https://github.com/dillonschultz93/quieto-tokens/issues). Include:

- What you expected to happen
- What actually happened
- Steps to reproduce (if applicable)
- Node version and OS
